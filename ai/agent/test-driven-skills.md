# 测试驱动的 Agent Skills 工程化构建

> **来源**：抖音 @AI启蒙小伙伴  
> **核心结论**：Skills 不是文档，而是可执行的专业知识——精简、可测试、从真实经验蒸馏。

---

## 核心观点

Skills 本质是**模块化的领域知识容器**，将通用 AI 转化为专业智能体。

首要约束是**上下文窗口为稀缺资源**——Skills 必须与系统提示、对话历史竞争有限空间。

---

## 四大核心原则

### 原则一：渐进式披露

| 层级 | 内容 | 加载时机 | 规模建议 |
|------|------|----------|----------|
| 元数据 | name, description, trigger | 始终加载 | ~100词 |
| SKILL.md | 核心流程、Gotchas | 触发后加载 | <500行 |
| references/ | schema、长模板、API文档 | 按需加载 | 无限制 |

**Gotchas**：那些违背直觉的具体陷阱（如 "users表软删除，查询必须带 WHERE deleted_at IS NULL"）

### 原则二：上下文预算意识

- ✅ 只补充 AI 不知的：项目特定约定、API 模式、边缘情况
- ❌ 避免空泛建议："适当处理错误"、通用常识

**检验标准**：每条指令须经得起"无此指令 AI 会做错吗"的检验。

### 原则三：测试驱动迭代

```
迭代流程：
1. 创建 8-10 个真实测试用例
2. 并行运行"有 Skill"vs"无 Skill"基线对比
3. 量化指标：token 消耗、准确率、耗时
4. 基于反馈迭代优化
5. 提炼通用方法，避免为过拟合堆砌规则
```

### 原则四：自由度匹配

| 任务类型 | 自由度 | 实现方式 |
|----------|--------|----------|
| 脆弱操作（如数据库迁移） | 低 | 特定脚本，严格约束 |
| 创意任务 | 高 | 启发式原则，开放探索 |
| 重复代码 | 固化 | 提取到 scripts/ 中复用 |

---

## Skill 设计检查清单

- [ ] description 写清楚触发条件
- [ ] 大文档放 references/ 而非 SKILL.md
- [ ] 包含具体的 Gotchas 而非泛泛建议
- [ ] 提供可执行的 scripts/ 脚本
- [ ] 建立测试用例验证效果

---

## 目录结构最佳实践

```
skill-name/
├── SKILL.md              # 精简核心流程 (<500行)
├── scripts/
│   ├── validate.py
│   └── generate.py
└── references/
    ├── api-docs.md
    ├── schema.json
    └── templates/
```
