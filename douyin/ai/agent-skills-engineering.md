# 测试驱动的 Agent Skills 工程化构建迭代指南

> **核心结论**：Skills 不是文档，而是可执行的专业知识——精简、可测试、从真实经验蒸馏。

---

## 🧠 核心观点

**Skills 本质是模块化的领域知识容器**，将通用 AI 转化为专业智能体。

其首要约束是**上下文窗口为稀缺资源**——Skills 必须与系统提示、对话历史竞争有限空间。

---

## 📊 四大核心原则

### 原则一：渐进式披露

采用**三层架构**设计 Skill 加载策略：

| 层级 | 内容 | 加载时机 | 规模建议 |
|------|------|----------|----------|
| 元数据 | name, description, trigger | 始终加载 | ~100词 |
| SKILL.md | 核心流程、Gotchas | 触发后加载 | <500行 |
| references/ | schema、长模板、API文档 | 按需加载 | 无限制 |

**关键实践**：
- 将 schema、长模板、API 文档移入 `references/`
- SKILL.md 仅保留核心流程与 **Gotchas**（那些违背直觉的具体陷阱）
- 示例："users表使用软删除，查询必须带 WHERE deleted_at IS NULL"

---

### 原则二：上下文预算意识

**默认 AI 已足够聪明**，只补充其不知的内容：

✅ **应该补充的**：
- 项目特定约定
- API 模式
- 边缘情况

❌ **避免的**：
- "适当处理错误"等空泛建议
- 通用编程常识
- 显而易见的提示

**检验标准**：每条指令须经得起"无此指令 AI 会做错吗"的检验。

---

### 原则三：测试驱动迭代

**量化验证是迭代的基础**：

```
迭代流程：
1. 创建 8-10 个真实测试用例
2. 并行运行"有 Skill"vs"无 Skill"基线对比
3. 量化指标：token 消耗、准确率、耗时
4. 基于反馈迭代优化
5. 提炼通用方法，避免为过拟合堆砌规则
```

**关键洞察**：测试不是一次性的，而是持续迭代的反馈循环。

---

### 原则四：自由度匹配

**根据任务类型调整约束强度**：

| 任务类型 | 自由度 | 实现方式 |
|----------|--------|----------|
| 脆弱操作（如数据库迁移） | 低 | 特定脚本，严格约束 |
| 创意任务 | 高 | 启发式原则，开放探索 |
| 重复代码 | 固化 | 提取到 scripts/ 中复用 |

**触发条件**：当测试显示 AI 反复重写相同代码时，将其固化为可重用脚本。

---

## 🎯 实践要点速查

### Skill 设计检查清单

- [ ] description 写清楚触发条件
- [ ] 大文档放 references/ 而非 SKILL.md
- [ ] 包含具体的 Gotchas 而非泛泛建议
- [ ] 提供可执行的 scripts/ 脚本
- [ ] 建立测试用例验证效果

### 目录结构最佳实践

```
skill-name/
├── SKILL.md              # 精简核心流程 (<500行)
├── scripts/
│   ├── validate.py       # 验证工具
│   └── generate.py       # 代码生成
└── references/
    ├── api-docs.md       # API 完整文档
    ├── schema.json       # 数据 schema
    └── templates/        # 长模板文件
```

---

## 💡 关键洞察

1. **上下文是稀缺资源**：每个 token 都要为最终效果负责
2. **测试驱动 > 经验驱动**：用数据说话，而非主观判断
3. **可执行 > 可读**：Skill 是代码的代码，不是用户手册
4. **迭代优于完美**：先跑通测试，再逐步优化
